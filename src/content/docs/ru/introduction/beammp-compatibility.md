NodeMP запускает серверные Lua-плагины, написанные для **BeamMP**, без изменения их кода.
Движок плагинов портирован из BeamMP (`TLuaEngine` / `TLuaPlugin`): тот же глобальный API
(`MP`, `Util`, `FS`, `Http`), те же имена и порядок событий, та же модель «одна папка
плагина → одно состояние Lua», наложение вето через `return 1`, `MP.RegisterEvent`,
`CreateEventTimer` и так далее.

Поверх этого расположен тонкий **слой трансляции** (`server/include/Lua/BeamMPCompat.h`),
который сглаживает единственное фундаментальное различие между двумя моделями:
идентификацию автомобилей.

## Проверено

Административный плагин **[CobaltEssentials](https://github.com/prestonelam2003/CobaltEssentials)**
v1.7.6 загружается и работает без изменений: инициализируются все модули и CobaltDB,
читается конфигурация, загружаются расширения и отвечают консольные команды (`ce help`,
`ce status`, `ce about`, `ce setgroup`, `ce setcfg`, …). Ноль ошибок Lua при загрузке и во
время работы.

## Что транслируется и как

### 1. Идентификаторы автомобилей

| | BeamMP | NodeMP |
|---|---|---|
| ID автомобиля | для каждого игрока: `(player_id, vehicle_id)` | один глобальный `vehicle_id` |
| Сигнатуры | `MP.RemoveVehicle(pid, vid)`, `MP.GetPositionRaw(pid, vid)` | `(vehicleId)` |
| Данные автомобиля | строка `"pid-vid:{ jbm, vcf, pos?, rot? }"` | объект `{ jbm, config, … }` |

Решение: **наш глобальный id выступает в роли `vehicle_id` из BeamMP** (он уникален и
корректно проходит в обе стороны, поэтому нет необходимости синтезировать счётчики для
каждого игрока).

- `MP.RemoveVehicle` и `MP.GetPositionRaw` (C++, `Lua/Engine.cpp`) принимают **как**
  `(vehicleId)`, **так и** `(playerId, vehicleId)` — в форме с двумя аргументами id является
  **вторым** аргументом (форма с одним аргументом использует свой единственный аргумент).
  Существующие вызовы с двумя аргументами по-прежнему разрешаются корректно, потому что id
  автомобиля, который хранит плагин, **и есть** глобальный id.
- `MP.GetPlayerVehicles(pid)` и аргумент `data` событий `onVehicleSpawn` /
  `onVehicleEdited` возвращаются в формате BeamMP `"pid-vid:{ … }"`, где `config`
  сопоставлен с `vcf` (`server/include/Lua/BeamMPCompat.h`). Поля `pos`/`rot` обычно
  **отсутствуют**: JSON автомобиля при спавне несёт `jbm` + `config` (без позиции верхнего
  уровня), а `onVehicleEdited` оборачивает сырой пакет правки, в котором нет полей автомобиля.
- `onVehicleReset` доставляется как обычный JSON (как и в BeamMP, где плагин вызывает
  `json.parse` напрямую); `onVehicleDeleted` несёт `(pid, vid)` и уже соответствует.

`NodeMP.*` (наш собственный API) **не изменён**: прелюдия `NodeMPApi.h` захватывает нативные
`MP.RegisterEvent` / `MP.GetPlayerVehicles` до того, как устанавливается шим, поэтому
переопределения в глобальной таблице `MP` затрагивают только плагины, которые вызывают
плоский API `MP` напрямую (то есть плагины BeamMP).

### 2. Консоль сервера — `onConsoleInput`

Ввод, набранный в консоли сервера, передаётся в Lua как событие `onConsoleInput(line)`
(`server/src/Core/ConsoleCommands.cpp`). Любая строка, которую **возвращает** обработчик,
выводится в консоль — именно так плагины BeamMP (например, CobaltEssentials с его командами
`ce …`) печатают свои ответы. Встроенные команды NodeMP (`help`, `kick`, `stop`, …)
продолжают работать; сообщение «неизвестная команда» подавляется, пока плагин слушает это
событие.

### 3. Конфигурация сервера — секция `[General]`

Многие плагины BeamMP читают `ServerConfig.toml` **напрямую** (прямой файловый ввод-вывод в
обход API), ожидая плоскую секцию `[General]` (`Name`, `Port`, `MaxCars`, `MaxPlayers`,
`Map`, `Private`, `Description`, `Tags`, `Debug`, `LogChat`, `ResourceFolder`, `AuthKey`).

При генерации `ServerConfig.toml` NodeMP теперь **также записывает зеркальную секцию
`[General]`** (`server/src/Core/Config.cpp`), синхронизированную с основными настройками.
Сам NodeMP её игнорирует (он читает собственные типизированные секции: `[Server]`,
`[Network]`, …).

:::note
Для **существующей** конфигурации, созданной более старой версией, добавьте секцию
`[General]` вручную (или удалите `ServerConfig.toml` и дайте серверу сгенерировать новую).
:::

## Установка плагина BeamMP

Распакуйте плагин в `Resources/Server/<PluginName>/` (точно так же, как в BeamMP). Файлы
`.lua` верхнего уровня загружаются автоматически в одно состояние Lua; вложенные модули
загружаются через `require` (в `package.path` добавляются оба пути — `<plugin>/?.lua` и
`<plugin>/lua/?.lua`).

## Известные ограничения

- **Упрощённая структура данных автомобиля.** Поля сопоставляются по смыслу
  (`vcf` ↔ `config`, `jbm`, `pos`, `rot`). Плагины, которые разбирают специфичные для
  BeamMP внутренние поля пакетов, могут работать с пониженной точностью.
- **Идентификаторы игроков.** `identifiers.ip` присутствует всегда; `identifiers.beammp`
  (ID форума BeamMP) — нет. Логика для отдельных аккаунтов, строго зависящая от `beammp`,
  может откатываться к имени или `ip`.
- **`[General]` в конфигурации** — это зеркало, а не источник истины: изменяйте значения в
  основных секциях (и поддерживайте `[General]` в синхронизации при ручном редактировании).
- Плагины с **клиентской** частью (скрипт-мод для BeamNG) требуют отдельной проверки —
  эта страница охватывает серверную сторону.
