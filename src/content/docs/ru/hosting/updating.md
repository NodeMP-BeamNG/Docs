Обновление сервера NodeMP — это в основном **замена бинарного файла**. Ваш `ServerConfig.toml`
и папки `Resources/` — это данные, которые читает сервер, а не часть исполняемого файла,
поэтому они переживают обновление нетронутыми.

## Что сохраняется

- **`ServerConfig.toml`** — сервер генерирует конфигурацию только когда её нет; он никогда не
  перезаписывает вашу. Новый бинарный файл продолжает читать ваш существующий файл.
- **`Resources/Server`** (плагины) и **`Resources/Client`** (моды) — остаются ровно такими,
  как есть.
- **Файлы данных плагинов**, записанные в собственной папке плагина (баны, сохранённый мир,
  конфигурация модулей и т. д.), — нетронуты.

Итак, безопасный порядок действий: остановить сервер, заменить бинарный файл, снова запустить.

## Шаги обновления

### systemd (Linux)

```bash
sudo systemctl stop nodemp-server
# replace the binary (rebuild via Docker, then copy — see Running the server):
sudo cp ./NodeMP-Server /opt/nodemp/NodeMP-Server
sudo chmod +x /opt/nodemp/NodeMP-Server
sudo systemctl start nodemp-server
journalctl -u nodemp-server -f
```

### Docker / docker-compose

Пересоберите образ и пересоздайте контейнер; ваш том `gamedata/` (конфигурация + ресурсы)
сохраняется:

```bash
cd deploy
docker compose up -d --build game-server
docker compose logs -f game-server
```

### Windows (нативно)

Остановите сервер, замените `NodeMP-Server.exe` в его папке и снова запустите. Расположенные
рядом `ServerConfig.toml` и `Resources/` сохраняются.

:::tip
Перед крупным обновлением сохраните копию `ServerConfig.toml` и `Resources/` (или всей папки
сервера), чтобы можно было мгновенно откатиться, если что-то пойдёт не так.
:::

## Новые ключи конфигурации после обновления

Если новая версия добавляет настройку, ваш существующий `ServerConfig.toml` просто не будет её
содержать, и сервер использует встроенное значение по умолчанию для этого ключа. Чтобы явно
подхватить только что добавленный ключ, либо добавьте его вручную в нужную секцию (см.
[Конфигурация](/ru/hosting/configuration/)), либо перегенерируйте конфигурацию заново:

1. Остановите сервер.
2. Переименуйте или сделайте резервную копию текущего `ServerConfig.toml`.
3. Запустите сервер один раз, чтобы сгенерировать новый файл со всеми текущими ключами.
4. Перенесите свои значения, затем перезапустите.

### Зеркало `[General]`

NodeMP добавляет секцию `[General]` в стиле BeamMP, которую некоторые серверные плагины читают
напрямую. В конфигурации, сгенерированной более **старой** версией, её может не быть. Если
плагин BeamMP ведёт себя некорректно после обновления, добавьте `[General]` вручную (или
перегенерируйте конфигурацию). Сам NodeMP игнорирует `[General]` — см.
[Совместимость с BeamMP](/ru/introduction/beammp-compatibility/).

## Напоминание об обновлении

Секция `[Updates]` управляет уведомлением «ваш сервер устарел»:

```toml
[Updates]
ImScaredOfUpdates  = false   # set true to mute the reminder
UpdateReminderTime = 360     # minutes between reminders
```

Оставьте `ImScaredOfUpdates = false`, чтобы действительно узнавать о новых релизах; увеличьте
`UpdateReminderTime`, если напоминание слишком назойливо.

## Следите за изменениями протокола и версий

NodeMP использует версионируемый **сетевой протокол** (сейчас **1.1**; см.
[Сетевой протокол](/plugins/protocol/)). Лаунчер и сервер согласуют версии во время
рукопожатия, а сервер сообщает **минимальную версию клиента** в своём маяке (`ClientMinVer`).
Несколько моментов, которые стоит помнить при обновлении:

- **Соответствуйте сети.** Когда релиз повышает версию протокола, лаунчеры/моды игроков и ваш
  сервер должны быть на совместимых версиях. Своевременное обновление сервера избавляет от
  отказа игрокам из-за несоответствия версий на рукопожатии.
- **Перепроверьте плагины.** Повышение версии протокола может изменить полезную нагрузку
  событий/автомобилей. Перепроверьте свои серверные плагины (и любые плагины BeamMP) после
  крупного обновления — начните с `Debug = true` в `[Logging]`, чтобы выявить ошибки Lua. См.
  [Lua API сервера](/plugins/server-api/) и [Перенос плагинов BeamMP](/ru/plugins/migrating/).
- **Перепроверьте клиентские моды.** Клиентские моды в `Resources/Client` автоматически
  перехешируются при изменении их файлов, поэтому достаточно заменить `.zip` — см.
  [Ресурсы и моды](/ru/hosting/resources/).
