Всё, что сервер добавляет поверх базовой игры, находится в одной папке, заданной ключом
`ResourceFolder` в `ServerConfig.toml` (по умолчанию **`Resources`**). У неё две половины:

```
Resources/
├── Server/     # server-side Lua plugins (run on the server)
└── Client/     # BeamNG mod .zip files (sent to every player)
```

Обе папки создаются пустыми при первом запуске, каждая с коротким `README.txt`. Разделение
важно: код в **`Server`** выполняется только на вашей машине; моды в **`Client`** скачиваются
и монтируются каждым подключившимся игроком.

## Серверные плагины — `Resources/Server`

Помещайте каждый плагин в **отдельную подпапку** здесь:

```
Resources/Server/
├── MyPlugin/
│   └── main.lua
└── CobaltEssentials/
    └── ...
```

Файлы `.lua` верхнего уровня в папке плагина загружаются в лексикографическом порядке и
**разделяют одно Lua-состояние** (один поток на папку плагина); вложенные модули загружаются
через `require`. Плагины могут использовать как совместимый с BeamMP глобальный API (`MP`,
`Util`, `FS`, `Http`), так и пространство имён `NodeMP.*` от NodeMP.

По умолчанию сервер пуст. В репозитории поставляется готовый набор примеров в
`server/examples/server-plugins` — пять взаимозависимых файлов `.lua` (`00_framework.lua`,
`10_admin.lua`, `20_dimensions.lua`, `30_persistence.lua`, `40_welcome.lua`), разделяющих одно
Lua-состояние. Скопируйте **всю папку `server-plugins`** в `Resources/Server/` (получив
`Resources/Server/server-plugins/`); тогда её файлы загрузятся вместе как **единый плагин** —
сначала `00_framework.lua` — а не как пять отдельных модулей. О том, как работают плагины, и о
полном API см.:

- [Обзор плагинов](/ru/plugins/overview/) — модель плагинов и их жизненный цикл.
- [Lua API сервера](/plugins/server-api/) — `MP.*` и `NodeMP.*`.
- [Совместимость с BeamMP](/ru/introduction/beammp-compatibility/) — запуск плагинов BeamMP
  без изменений.

:::note
Плагин читает и записывает файлы данных **внутри собственной папки** (в песочнице) — именно
так примеры плагинов admin/persistence хранят баны, сохранённый мир и конфигурацию модулей.
:::

## Клиентские моды — `Resources/Client`

Кладите сюда файлы модов BeamNG в формате **`.zip`**, и они автоматически раздаются каждому
игроку:

```
Resources/Client/
├── my_map.zip
├── cool_car.zip
└── mods.json        # auto-generated (see below)
```

Когда сервер (пере)сканирует эту папку, он:

1. Хеширует каждый `.zip` алгоритмом **SHA-256** и записывает его размер.
2. Сообщает список модов подключающимся клиентам.
3. Лаунчер **скачивает и монтирует** каждый мод для каждого игрока при подключении, а затем
   **размонтирует** его при выходе — так игрокам не нужно устанавливать вашу карту или пак
   автомобилей вручную.

Модами считаются только файлы `.zip`; файлы `README.txt` / `.md` пропускаются, а всё, что не
является zip, игнорируется с предупреждением.

### `mods.json`

`mods.json` — это **автоматически создаваемый кэш**, который сервер поддерживает в
`Resources/Client`; вы не пишете его вручную. Для каждого файла мода он хранит время последней
записи, размер, хеш SHA-256 и флаг `protected`, чтобы сервер мог пропускать перехеширование
неизменившихся файлов и перехешировать только те, что вы добавили или изменили. Записи для
удалённых файлов удаляются автоматически.

Это значит, что обновление клиентского мода — это просто **замена `.zip`**: сервер замечает
изменившийся размер/метку времени и перехеширует его при следующем сканировании. (На свежем
пустом сервере `mods.json` может содержать лишь `null`, пока первое сканирование не заполнит
его.)

## Указание другого расположения через `ResourceFolder`

Чтобы хранить ресурсы вне рабочего каталога, задайте `ResourceFolder` другой путь:

```toml
[Server]
ResourceFolder = "/srv/nodemp-res"
```

Сервер ожидает те же подпапки `Server/` и `Client/` по любому выбранному вами пути и создаёт
их, если они отсутствуют.
