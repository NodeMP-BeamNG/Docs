Сервер NodeMP появляется в **публичном браузере серверов** только если он зарегистрирован в
бэкенде. Регистрация даёт вашему серверу **идентификатор хоста (host id)** и **секрет хоста
(host secret)**; с ними он открывает аутентифицированную *хост-сессию* и отправляет
периодический *маяк (beacon)*, который поддерживает его в списке.

Если вам нужен только приватный сервер или сервер для локальной сети (LAN), всё это можно
пропустить — см. [Тестирование без бэкенда](#тестирование-без-бэкенда) внизу страницы.

## Как работает попадание в список

```
register host ──▶ HostId + HostSecret ──▶ ServerConfig.toml [Backend]
                                              │
server starts ───────────────────────────────┘
   │  POST /v1/hosts/session   (HostId + HostSecret)  → session token
   └─ POST /v1/hosts/beacon    (every interval)       → stays listed
```

Сервер обращается к бэкенду по URL из `NODEMP_BACKEND_URL` (см.
[Запуск сервера](/ru/hosting/running/)). Он открывает хост-сессию с вашими учётными данными,
затем маяком сообщает своё текущее состояние (число игроков, карта, имя, теги, версия, список
модов…), чтобы браузер поддерживал свежую запись *присутствия (presence)*, которая истекает
вскоре после прекращения маяков.

## 1. Получите HostId и HostSecret

Сначала вам нужен аккаунт в бэкенде (зарегистрируйтесь через лаунчер или
`POST /v1/auth/register`). Затем создайте хост. Проще всего использовать вспомогательный
скрипт `deploy/register-host.sh`, который выполняет вход, вызывает `POST /v1/hosts`, выводит
учётные данные и может записать их прямо в вашу конфигурацию:

```bash
NODEMP_BACKEND_URL=http://<backend-ip>:8080 \
  bash deploy/register-host.sh http://<backend-ip>:8080 /opt/nodemp/ServerConfig.toml
```

Он выводит:

```
HostId     = <id>
HostSecret = <secret>
(the secret is shown only once — store it now)
```

:::caution
**Секрет хоста показывается лишь один раз**. Сохраните его немедленно. Если вы его потеряете,
смените его через `POST /v1/hosts/{id}/rotate-secret` (он возвращает новый секрет и аннулирует
старый).
:::

Предпочитаете сделать это вручную? Создайте хост напрямую:

```bash
curl -X POST http://<backend-ip>:8080/v1/hosts \
  -H "Authorization: Bearer <access-token>" \
  -H 'Content-Type: application/json' \
  -d '{"label":"my-nodemp-host"}'
# → {"host_id":"...","host_secret":"..."}
```

## 2. Внесите учётные данные в конфигурацию

Добавьте их в секцию `[Backend]` файла `ServerConfig.toml` и установите `Private = false`,
чтобы узел был допущен в публичный список:

```toml
[Server]
Private = false

[Backend]
HostId     = "<id>"
HostSecret = "<secret>"
```

Перезапустите сервер. При успешном запуске он сообщает о сети как *registered as '<id>'*.

## 3. Выберите: приватный по IP или в публичном списке

Два флага бэкенда определяют, насколько строгим будет путь подключения. Выберите режим,
соответствующий вашей цели:

| | Приватный / по IP (по умолчанию) | Публичный список |
|---|---|---|
| Как игроки находят сервер | по прямому IP в лаунчере | в браузере серверов |
| `NODEMP_PROBE_ENABLED` | `false` | `true` (бэкенд должен достучаться до `30814`) |
| `NODEMP_STRICT_REDEEM_IP` | `false` (гибко за NAT) | `true` (строже) |
| `ServerConfig.toml` `Private` | `true` | `false` |
| Порт `30814` открыт | для игроков | для игроков **и** пробы |

Аккаунты работают в обоих режимах. Для публичного размещения **проба** бэкенда должна иметь
возможность достучаться до вашего `30814` извне, поэтому пробросьте порт соответствующим
образом.

## 4. Предполётная проверка цепочки

Прежде чем приглашать игроков, проверьте весь путь аутентификации — бэкенд доступен →
хост-сессия → билет на подключение → погашение — с помощью `deploy/preflight.sh`. Запускайте
его **на хосте игрового сервера**, чтобы бэкенд видел тот же публичный IP:

```bash
NODEMP_BACKEND_URL=http://<backend-ip>:8080 \
NODEMP_HOST_ID=<id> NODEMP_HOST_SECRET=<secret> \
NODEMP_HOST_ADDR=<public-ip>:30814 \
  bash deploy/preflight.sh
```

Каждый шаг выводит `OK`/`FAIL` с подсказкой, поэтому сломанное звено (адрес, JWT, учётные
данные хоста, проба, несоответствие IP/хоста) становится очевидным ещё до того, как игроки
попытаются подключиться.

## Справочник по эндпоинтам хоста

Бэкенд предоставляет следующие маршруты `/v1` (сервер и вспомогательные скрипты вызывают их за
вас):

| Метод и путь | Авторизация | Назначение |
|---|---|---|
| `POST /v1/hosts` | аккаунт | Создать хост → `{ host_id, host_secret }`. |
| `GET /v1/hosts` | аккаунт | Список ваших хостов. |
| `POST /v1/hosts/{id}/rotate-secret` | аккаунт | Выдать новый секрет → `{ host_secret }`. |
| `DELETE /v1/hosts/{id}` | аккаунт | Удалить хост. |
| `POST /v1/hosts/session` | учётные данные хоста | Открыть хост-сессию → `{ session_token, expires_in, beacon_interval }`. |
| `POST /v1/hosts/beacon` | хост-сессия | Обновить присутствие → `{ listed, advisories, next_in }`. |

Каждый аккаунт ограничен максимальным числом хостов (у администраторов без ограничений).

## Тестирование без бэкенда

Для локальной проверки или проверки в LAN всё это **не** нужно:

- **Loopback** — подключение с `127.0.0.1` доверяется автоматически, когда
  `[Network] TrustLoopback = true` (по умолчанию), поэтому вы можете подключиться к
  собственному серверу без бэкенда и без аккаунта. Это путь
  [быстрого старта](/ru/hosting/quick-start/).
- **LAN** — оставьте `[Backend]` пустым и `Private = true`; игроки подключаются по LAN-адресу
  вашей машины и прямому порту. Учтите, что игроки с других машин **не** являются loopback,
  поэтому они всё равно проходят обычный путь аккаунт/билет, если сервер указывает на бэкенд;
  для чистой проверки в LAN без аккаунтов оставьте `NODEMP_BACKEND_URL`
  незаданным/недоступным и используйте loopback на самом хосте.

:::caution
Если ваш сервер находится за обратным прокси, который подключается с `127.0.0.1`, установите
`TrustLoopback = false` — иначе внешние игроки, приходящие через прокси, могут быть восприняты
как доверенные loopback-пиры.
:::

## Частые проблемы

- **Нет в списке** — включён ли публичный режим (`Private = false`,
  `NODEMP_PROBE_ENABLED=true`) и достижим ли порт `30814` извне? Если проба не может до вас
  достучаться, вас не разместят в списке.
- **`redeem failed` / `ticket_host_mismatch`** — адрес хоста в билете (публичный IP) не совпал
  с IP, с которого сервер открыл свою сессию (NAT). Для нестрогой проверки установите
  `NODEMP_STRICT_REDEEM_IP=false`.
- **Не удаётся открыть хост-сессию** — неверные `HostId`/`HostSecret`. Перезапустите
  `register-host.sh` или смените секрет.
