Это сетевой протокол, на котором общаются лаунчер/мод и сервер NodeMP. Авторы плагинов редко
касаются его напрямую — поверх него лежат [Lua API сервера](/ru/plugins/server-api/) и
[API клиентского мода](/ru/plugins/client-api/) — но он является источником истины для
идентификаторов автомобилей, событий и семантики пакетов.

> **Версия:** 1.1  
> **Дата:** 2026-06-21  
> **Статус:** модель глобального `VehicleID` + постоянный владелец синхронизации  

## 1. Цели и принципы проектирования

1. **Систематическая идентификация типов** — каждый пакет идентифицируется единственным идентификатором типа `uint8`. Устаревшие случайные однобуквенные коды (`Z`, `O`, `C`, `E`, …) упразднены.
2. **Чёткое разделение каналов** —
   * **UDP** — высокочастотное ненадёжное состояние: трансформация автомобиля (позиция, поворот, скорость, ввод). Потери допустимы; сервер отсеивает ретрансляцию по пространственному признаку.
   * **TCP** — всё остальное: диффы, жизненный цикл, занятость мест, заморозка/состояние, чат, игроки, события, передача файлов.
   * Лаунчер (BeamMP-Launcher) применяет это разделение автоматически: пакеты, у которых первый байт не `O`, `T`, `N`, `W`, `Y`, `V`, `E` или `C` и размер ≤ 1000 bytes, идут по UDP; всё остальное — по TCP.
3. **VehicleID** — единый **глобальный** идентификатор, уникальный в пределах сервера и выделяемый монотонно (он **не** кодирует владельца). Во всех сетевых полезных нагрузках он передаётся как десятичная **строка** (например, `"42"`), чтобы избежать проблем с точностью IEEE-754 и сохранить стабильность ключей таблиц. Владение/атрибуция (`spawner`) и текущее полномочие синхронизации (`syncOwner`) передаются как отдельные поля. Автомобили **постоянны**: они не уничтожаются при отключении игрока.
4. **Компактный JSON** — JSON сохраняется для неструктурированных/динамических данных (электрика, контроллеры, конфигурации автомобилей, события). Ключи однобуквенные или очень короткие. Там, где схема фиксирована, массивы предпочтительнее объектов.
5. **State-diff (диффы состояния)** — пакеты электрики и контроллеров содержат *только* поля, изменившиеся с момента последнего подтверждённого состояния.
6. **Путь обратной миграции** — спецификация явно сопоставляет устаревшие коды новым идентификаторам, чтобы рефакторинг можно было выполнять поэтапно.

## 2. Кадрирование транспорта

### 2.1 TCP
```
[int32  length] [uint8 type_id] [payload_body ...]
```
* `length` — общий размер всего, что идёт **после** самого поля длины (`type_id + payload_body`).
* `type_id` — один из идентификаторов, определённых в §4.
* `payload_body` — JSON (UTF-8) для всех типов пакетов, **кроме** чанков файлового потока, которые могут нести сырые двоичные данные после небольшого JSON-заголовка (см. §4.6).
* **Сжатие** — полезные нагрузки больше 400 bytes МОГУТ сжиматься zlib. При сжатии полезная нагрузка (после `type_id`) начинается с ASCII-маркера `ABG:`, за которым следует поток zlib. `type_id` располагается *перед* сжатой обёрткой. Для логического протокола это полностью прозрачно.

### 2.2 UDP
UDP-датаграммы предваряются `PlayerID + 1` отправителя и двоеточием для привязки эндпоинта. Лаунчер добавляет этот префикс автоматически; ядро сервера удаляет его перед разбором.

```
[uint8 player_id_plus_1] ':' [uint8 type_id] [payload_body ...]
```

* `player_id_plus_1` — префикс транспортного уровня, добавляемый лаунчером. `PlayerID = byte - 1`.
* `type_id` для UDP **всегда** в диапазоне `0x00–0x0F`.
* `payload_body` — компактный JSON. Внутри UDP **нет** префикса длины; полезная нагрузка тянется до конца датаграммы.
* **Сжатие** — то же правило `ABG:` + zlib, что и для TCP, применяется к `payload_body`, когда он превышает 400 bytes.
* **Привязка канала** — сервер ДОЛЖЕН доверять `player_id_plus_1` только после завершения UDP-рукопожатия авторизации лаунчера (см. §9).

|--------------|-------------|-------|

## 4. Справочник типов пакетов

### 4.1 `0x00 – 0x0F` — состояние UDP

> **Канал:** только UDP  
> **Частота:** до 60 Hz на автомобиль  

#### `0x01` — VehicleTransform
* **Направление:** клиент → сервер → релевантные клиенты (пространственный отсев)
* **Канал:** только UDP
* **Когда:** каждый тик симуляции для каждого активного автомобиля, принадлежащего клиенту. Частота зависит от клиента (исторически ~50 Hz).
* **Полезная нагрузка (компактный JSON):**
  ```json
  {"v":"42","p":[1234.56,78.90,12.34],"sp":[1234.60,78.95,12.30],"r":[0.0,0.0,0.7071,0.7071],"vl":[5.2,0.0,-1.1],"vr":[0.0,0.0,0.5],"tim":12.3,"ping":0.05}
  ```
* **Схема:**
  * `v` — `VehicleID` (строка глобального идентификатора)
  * `p` — позиция `[x, y, z]` (массив float). Позиция центра тяжести (COG); используется как **запасной вариант**, когда `sp` отсутствует.
  * `sp` — позиция опорного узла (ref-node) `[x, y, z]` (массив float). **Предпочтительна** — точка, относительно которой движок спавнит автомобиль, поэтому сервер запекает её в повторно передаваемые спавны. Необязательна: старые клиенты отправляют только `p`.
  * `r` — кватернион поворота `[x, y, z, w]` (массив float)
  * `vl` — линейная скорость `[x, y, z]` (массив float)
  * `vr` — угловая скорость `[x, y, z]` (массив float)
  * `tim` — локальное время симуляции отправителя (float, секунды), используется для интерполяции/предсказания.
  * `ping` — собственный пинг отправителя (float, секунды), используется для оценки смещения времени.
* **Замечание по проектированию:** в UDP нет дельта-кодирования; клиент отправляет абсолютную трансформацию каждый тик. **Ввод водителя здесь НЕ передаётся** — он кодируется диффами и должен доставляться надёжно, поэтому идёт в `0x10` (ключ `i`). Сервер может отбросить или отсеять ретрансляцию исходя из пространственной релевантности.

#### `0x02` — VehicleTransformBatch *(зарезервировано)*
* Зарезервировано для будущего пакетного двоичного формата, упаковывающего несколько автомобилей в одну датаграмму. В v1.0 не используется.

#### `0x00`, `0x03–0x0F` — зарезервировано

### 4.2 `0x10 – 0x1F` — диффы свойств автомобиля

> **Канал:** только TCP  
> **Семантика:** State-diff. Клиенты и серверы хранят *последний известный* снимок для каждого автомобиля. Передаются только пары ключ/значение, отличающиеся от последнего снимка. Получатель сливает дифф в свой локальный снимок.

#### `0x10` — VehicleElectricsDiff
* **Направление:** клиент → сервер → релевантные клиенты
* **Канал:** только TCP
* **Когда:** при любом изменении значения электрики для принадлежащего автомобиля.
* **Полезная нагрузка:**
  ```json
  {"v":"42","d":{"lights_state":1,"signal_left_input":1},"i":{"s":-0.3,"t":0.75,"b":0.0,"c":0.0,"p":0.0,"g":"D1"}}
  ```
* **Схема:**
  * `v` — `VehicleID` (строка глобального идентификатора)
  * `d` — объект, содержащий только изменившиеся ключи **электрики** (фары, сигналы, топливо и т. д.). Не структурирован и различается для каждого мода автомобиля; неизвестные ключи пересылаются дословно.
  * `i` — объект, содержащий только изменившийся **ввод водителя** (отправляется здесь, а не в `0x01`, потому что диффы должны быть надёжными):
    * `s` — руль `[-1..1]`
    * `t` — газ `[0..1]`
    * `b` — тормоз `[0..1]`
    * `c` — сцепление `[0..1]`
    * `p` — стояночный тормоз `[0..1]`
    * `g` — передача (строка или int; периодически пересылается, чтобы автомобиль, заспавненный на передаче, совпадал)
* **Замечание по проектированию:** `d` и `i` независимы, и может присутствовать любой из них. Сервер пересылает дифф дословно и (начиная с v1.1) сливает его в последний известный снимок для каждого автомобиля, чтобы позже подключающимся клиентам можно было повторно отправить текущее состояние (см. §5.1).

#### `0x11` — VehicleControllersDiff (глубокое состояние автомобиля)
* **Направление:** клиент → сервер → релевантные клиенты
* **Канал:** только TCP
* **Когда:** при любом изменении глубокого состояния автомобиля (вызов контроллера, режим трансмиссии, двигатель/зажигание, разрыв узла, присоединение/отсоединение сцепки, огонь).
* **Полезная нагрузка:** мультиплексированный объект с ключами по подсистемам; может присутствовать любое подмножество ключей.
  ```json
  {"v":"42","c":{"controllerName":"...","functionName":"...","variables":[...]},
   "l":{...},"e":{...},"g":[...],"n":{...},"cp":[...],"f":{"on":1,"n":[12,34]}}
  ```
* **Схема:**
  * `v` — `VehicleID` (строка глобального идентификатора)
  * `c` — один вызов контроллера (controllerName/functionName/variables), см. синхронизацию контроллеров.
  * `l` — изменившиеся режимы устройств трансмиссии (дифференциал/блокировка/раздаточная коробка).
  * `e` — состояние двигателя (уровень зажигания, зажигание/стартер/глохнущий ДВС).
  * `g` / `n` — сломанные группы разрушения / снимок узлов.
  * `cp` — присоединение/отсоединение сцепки.
  * `f` — состояние огня автомобиля: `{on:0|1, n:[cid,...]}` (идентификаторы горящих узлов) или `{on:1, all:1}` (полностью в огне). Воспроизводится для поздно подключившихся, пока горит; сбрасывается при тушении.

#### `0x12` — VehicleGearboxDiff *(зарезервировано)*
* Зарезервировано для явного состояния коробки передач/трансмиссии, если диффа электрики/контроллеров окажется недостаточно.

#### `0x13` — VehicleDamageDiff *(зарезервировано)*
* Зарезервировано для дельт повреждений узлов/балок.

#### `0x14–0x1F` — зарезервировано

### 4.3 `0x20 – 0x2F` — занятость автомобиля

> **Канал:** только TCP

#### `0x20` — VehicleEnter
* **Направление:** клиент → сервер → все клиенты
* **Канал:** только TCP
* **Когда:** игрок садится на место в автомобиле.
* **Полезная нагрузка:**
  ```json
  {"v":"42","p":5,"s":"driver"}
  ```
* **Схема:**
  * `v` — `VehicleID` (строка глобального идентификатора)
  * `p` — `PlayerID` входящего
  * `s` — имя места
* **Замечание по проектированию:** занятие места **водителя** делает этого игрока владельцем синхронизации автомобиля; сервер пересчитывает `pickSyncOwner` и, если он изменился, рассылает `0x35`. Состояние блокировки проверяется по политике **спавнера**.

#### `0x21` — VehicleExit
* **Направление:** клиент → сервер → все клиенты
* **Канал:** только TCP
* **Когда:** игрок выходит из автомобиля.
* **Полезная нагрузка:**
  ```json
  {"v":"42","p":5}
  ```
* **Замечание по проектированию:** когда **водитель** выходит, сервер пересчитывает `pickSyncOwner` (спавнер, если онлайн, иначе наименее загруженный) и рассылает `0x35`, если владелец синхронизации изменился.

#### `0x22` — VehicleSeatChange
* **Направление:** клиент → сервер → все клиенты
* **Канал:** только TCP
* **Когда:** игрок пересаживается на другое место в том же автомобиле.
* **Полезная нагрузка:**
  ```json
  {"v":"42","p":5,"s":"passenger"}
  ```

#### `0x23–0x2F` — зарезервировано

### 4.4 `0x30 – 0x3F` — жизненный цикл автомобиля

> **Канал:** только TCP  
> **Надёжность:** критично — эти пакеты ДОЛЖНЫ доставляться и подтверждаться.

#### `0x30` — VehicleSpawn
* **Направление:**
  * клиент → сервер: запрос
  * сервер → все: рассылка
* **Канал:** только TCP
* **Когда:** игрок спавнит новый автомобиль из гаража или через Lua.
* **Полезная нагрузка запроса клиента:**
  ```json
  {"car":{"jbm":"etk800","config":"..."},"cid":7}
  ```
  * `car` — полный JSON конфигурации автомобиля (та же форма, что и устаревшие данные спавна).
  * `cid` — необязательный корреляционный идентификатор клиента (монотонный в рамках сессии). Сервер возвращает его без изменений в рассылке, чтобы спавнер сопоставлял собственное эхо по идентификатору, а не по порядку прибытия.
* **Полезная нагрузка рассылки сервера:**
  ```json
  {"v":"42","o":5,"so":5,"n":"PlayerName","r":"SCR","car":{"jbm":"etk800","config":"..."},"cid":7,"h":2463534242,"lm":0,"u":42}
  ```
  * `v` — строка глобального `VehicleID`, назначенная сервером.
  * `o` — `PlayerID` спавнера (атрибуция; может устареть после переподключения)
  * `so` — текущий `PlayerID` владельца синхронизации (на момент спавна это спавнер; `-1` только если сервер в остальном пуст)
  * `n` — отображаемое имя спавнера
  * `r` — тег роли спавнера (пусто, если спавнер офлайн, например во время синхронизации при подключении)
  * `car` — полный JSON конфигурации автомобиля
  * `cid` — присутствует только если исходный клиент его предоставил; используется только спавнером.
  * `h` — вычисленный сервером хеш конфигурации (FNV-1a по сохранённой конфигурации). Присутствует в подтверждённых спавнах и при синхронизации подключения (§9). Клиенты хранят его и сравнивают с манифестом (`0x36`) для обнаружения расхождения конфигурации.
  * `lm` — режим блокировки (`0` открыт / `1` только пассажиры / `2` закрыт). См. `0x39`.
  * `u` — **статический** идентификатор аккаунта владельца (постоянный; `-1` для гостевых/loopback-владельцев). Отличается от `o`, который является идентификатором в рамках сессии.
* **Замечание по проектированию:** сервер ДОЛЖЕН проверить лимиты спавна и Lua-хуки (`onVehicleSpawn`) перед рассылкой. При отклонении сервер отправляет `0x31` VehicleDelete только запрашивающему клиенту (`cid` всё равно возвращается, чтобы клиент очистил свою ожидающую запись). Тот же пакет воспроизводится для каждого существующего автомобиля во время начальной синхронизации подключающегося клиента (§9), поэтому клиент может получить спавн, у которого `o` равен его собственному идентификатору; при отсутствии соответствующего ожидающего `cid` он ДОЛЖЕН рассматривать это как удалённый автомобиль.

#### `0x31` — VehicleDelete
* **Направление:** клиент → сервер → все клиенты
* **Канал:** только TCP
* **Когда:** игрок удаляет свой автомобиль, или сервер отклоняет спавн / принудительно удаляет.
* **Полезная нагрузка:**
  ```json
  {"v":"42","o":5}
  ```
  * `o` необязателен в направлении клиент→сервер (сервер выводит его из автомобиля), но обязателен в рассылках сервера (он несёт идентификатор спавнера).
* **Замечание по проектированию:** удаление авторитетно и явно — удалить автомобиль (`0x31`) может только его текущий **владелец синхронизации** или его **спавнер**, либо серверный Lua через `NodeMP.vehicles.remove(vehicleId)`. Автомобили постоянны: отключение игрока **НЕ** удаляет его автомобили (см. §7).

#### `0x32` — VehicleReset
* **Направление:** клиент → сервер → все клиенты
* **Канал:** только TCP
* **Когда:** игрок сбрасывает автомобиль (ремонт / возрождение на последней позиции на дороге).
* **Полезная нагрузка:**
  ```json
  {"v":"42","o":5,"d":{"pos":[0.0,0.0,0.0],"rot":[0.0,0.0,0.0,1.0]}}
  ```
  * `v` — `VehicleID` (строка глобального идентификатора)
  * `d` — необязательный JSON состояния сброса (например, начальная трансформация). Пустой `d` означает сброс по умолчанию.

#### `0x33` — VehiclePaintChange
* **Направление:** клиент → сервер → все клиенты
* **Канал:** только TCP
* **Когда:** игрок меняет покраску или конфигурацию материалов автомобиля.
* **Полезная нагрузка:**
  ```json
  {"v":"42","o":5,"p":[{"base":"#FF0000","roughness":0.5}]}
  ```
  * `v` — `VehicleID` (строка глобального идентификатора)
  * `p` — массив покраски (та же схема, что и тело устаревшего `Op`).
  * В **рассылке сервера** после сохранения покраски добавляется поле `h` (обновлённый хеш конфигурации), чтобы получатели обновили свой сохранённый хеш и манифест оставался согласованным.

#### `0x34` — VehicleConfigChange
* **Направление:** клиент → сервер → все клиенты
* **Канал:** только TCP
* **Когда:** игрок редактирует существующий автомобиль (замена деталей, тюнинг и т. д.).
* **Полезная нагрузка:**
  ```json
  {"v":"42","o":5,"d":{"parts":{"etk800_body":"etk800_body_sedan"}}}
  ```
  * `v` — `VehicleID` (строка глобального идентификатора)
  * `d` — JSON диффа или полной конфигурации, в зависимости от реализации клиента. Сервер считает его новой авторитетной конфигурацией и обновляет свой реестр.
  * В **рассылке сервера** после применения конфигурации добавляется поле `h` (обновлённый хеш конфигурации), чтобы получатели обновили свой сохранённый хеш.

#### `0x35` — VehicleSyncOwnerChange
* **Направление:** сервер → все клиенты
* **Канал:** только TCP
* **Когда:** сервер (пере)назначает полномочие синхронизации автомобиля — при спавне, при входе/выходе водителя, при отключении текущего владельца синхронизации или при балансировке нагрузки, когда клиент подключается/уходит.
* **Полезная нагрузка:**
  ```json
  {"v":"42","so":7}
  ```
  * `v` — `VehicleID` (строка глобального идентификатора)
  * `so` — новый `PlayerID` владельца синхронизации (`-1` = нет; только когда у сервера нет подходящих клиентов).
* **Замечание по проектированию:** клиент, ставший владельцем синхронизации, начинает отправлять состояние для этого автомобиля (профили синхронизации см. в §5.3) и переключает локальный автомобиль в роль «local»; предыдущий владелец прекращает и переключает его в «remote». Это единственный пакет, меняющий, какой клиент авторитетен для автомобиля.

#### `0x36` — VehicleManifest
* **Направление:** сервер → все клиенты
* **Канал:** только TCP
* **Когда:** периодически (переменная окружения `NODEMP_MANIFEST_INTERVAL_S`, по умолчанию 15s; `0` отключает) каждому синхронизированному клиенту. Это авторитетный снимок множества автомобилей мира, используемый для клиентской сверки.
* **Полезная нагрузка:**
  ```json
  {"veh":[{"v":"42","o":5,"so":7,"h":2463534242}]}
  ```
  * `veh` — массив всех автомобилей (кроме unicycle): `v` глобальный идентификатор, `o` спавнер, `so` владелец синхронизации, `h` хеш конфигурации.
* **Замечание по проектированию:** при получении (только в полностью синхронизированном состоянии) клиент выполняет сверку: автомобили, которые есть в манифесте, но отсутствуют локально, подтягиваются через `0x37`; локальные автомобили, отсутствующие в манифесте, удаляются (пропущенный `0x31`); отличающийся `so` применяет смену владельца синхронизации; отличающийся `h` (только для автомобилей, которыми клиент **не** владеет) запускает удаление с последующим `0x37` для повторного получения конфигурации. Владелец синхронизации никогда не пересинхронизирует собственный автомобиль (он никогда не получает эхо собственных `0x33`/`0x34`, поэтому его сохранённый хеш намеренно устаревший).

#### `0x37` — VehicleResyncRequest
* **Направление:** клиент → сервер
* **Канал:** только TCP
* **Когда:** сверке манифеста клиента требуется полное состояние одного автомобиля (он отсутствует локально или его конфигурация разошлась).
* **Полезная нагрузка:**
  ```json
  {"v":"42"}
  ```
* **Замечание по проектированию:** сервер отвечает, повторно передавая этот единственный автомобиль только запросившему — спавн `0x30`, за которым следует воспроизведение его накопленного динамического состояния (`0x10`/`0x11`), идентично шагу для одного автомобиля из синхронизации подключения (§9). Если автомобиль больше не существует, сервер игнорирует запрос (вместо этого сверка клиента удаляет его локально). Клиенты ограничивают частоту повторных запросов по каждому автомобилю.

#### `0x38` — VehicleStateBlob
* **Направление:** владелец синхронизации → сервер → другие клиенты (и сервер → подключающийся во время синхронизации)
* **Канал:** только TCP
* **Когда:** владелец синхронизации периодически захватывает **полное физическое состояние** автомобиля и выгружает его; сервер сохраняет его и воспроизводит при подключении/пересинхронизации. Именно это позволяет повреждённому автомобилю сохранить повреждения (а также деформацию, состояние деталей, топливо, коробку передач, гидравлику) после отключения его владельца.
* **Полезная нагрузка:**
  ```json
  {"v":"42","s":"<base64 state blob>"}
  ```
  * `v` — глобальный идентификатор автомобиля.
  * `s` — непрозрачный, специфичный для движка base64-снимок, создаваемый модулем автомобиля `statepack` (сломанные/деформированные балки, позиции узлов, гидравлика, ввод водителя, торсионы, обороты двигателя, коробка передач, плюс метаданные топлива и состояния деталей).
* **Замечание по проектированию:** устанавливать blob может только текущий владелец синхронизации; сервер отклоняет остальных и ограничивает blob 4 MiB (типичные blob'ы — несколько сотен KB; в сети применяется сжатие zlib). Владелец отправляет не чаще, чем примерно раз в ~2 s, и только когда blob действительно изменился (целый, неизменный автомобиль не стоит ничего). На принимающей стороне снимок применяется **после** грубого воспроизведения `0x10`/`0x11`, поэтому имеет приоритет. Позиции узлов включаются, только когда автомобиль действительно повреждён. Во время синхронизации подключения blob буферизуется клиентом, пока не приземлится спавн автомобиля (поставленный в очередь применения), после чего применяется — см. §9.

#### `0x39` — VehicleLockChange
* **Направление:** владелец → сервер → все клиенты
* **Канал:** только TCP
* **Когда:** владелец меняет политику доступа к автомобилю (кто может им управлять / входить в него). Сервер хранит политику на автомобиле (она сохраняется после ухода владельца) и повторно рассылает её.
* **Полезная нагрузка:**
  ```json
  {"v":"42","m":1}
  ```
  * `v` — глобальный идентификатор автомобиля.
  * `m` — режим блокировки: `0` = открыт (любой может управлять/ехать), `1` = только пассажиры (управлять может только владелец; другие могут занимать пассажирские места — без угона), `2` = закрыт (входить может только владелец).
* **Замечание по проектированию:** сервер принимает изменение только от **владельца** автомобиля — его текущей сессии спавнера или переподключившегося владельца, сопоставленного по статическому идентификатору аккаунта (см. §5.3). Оно применяется в обработчиках занятости (`0x20` вход и `0x22` смена места): не-владелец не допускается на место водителя в режиме 1 и на любое место в режиме 2. Владелец обходит блокировку. Режим также передаётся в пакете спавна (`lm`), чтобы подключающиеся узнавали его, не дожидаясь `0x39`.

#### `0x3A–0x3F` — зарезервировано

### 4.5 `0x40 – 0x4F` — зарезервировано (была заморозка / состояние)

> **Статус: НЕ РЕАЛИЗОВАНО — зарезервировано.** Сервер сегодня не отправляет эти пакеты
> (`PacketTypes.h` помечает `0x40-0x4F` как зарезервированные). Приведённая ниже схема
> заморозки / LOD сохранена для справки на случай возможной будущей реализации; клиенты не
> должны ожидать её сейчас.
> **Канал:** только TCP · **Семантика:** генерируются сервером; клиенты их не отправляют.

#### `0x40` — VehicleFreeze
* **Направление:** сервер → клиент (для конкретного наблюдателя)
* **Канал:** только TCP
* **Когда:** автомобиль переходит в состояние `Frozen` для конкретного наблюдателя, потому что расстояние > 500 m.
* **Полезная нагрузка:**
  ```json
  {"v":"42"}
  ```
  * `v` — `VehicleID` (строка глобального идентификатора)

#### `0x41` — VehicleUnfreeze
* **Направление:** сервер → клиент (для конкретного наблюдателя)
* **Канал:** только TCP
* **Когда:** автомобиль выходит из состояния `Frozen` для конкретного наблюдателя.
* **Полезная нагрузка:**
  ```json
  {"v":"42"}
  ```
  * `v` — `VehicleID` (строка глобального идентификатора)
* **Замечание по проектированию:** после отправки `0x41` сервер ДОЛЖЕН немедленно отправить `0x42` VehicleSaveState, чтобы клиент мог восстановить физическое состояние.

#### `0x42` — VehicleSaveState
* **Направление:** сервер → клиент (для конкретного наблюдателя)
* **Канал:** только TCP
* **Когда:** сразу после `0x41`, чтобы предоставить полный снимок замороженного состояния (~10–13 KB).
* **Полезная нагрузка:**
  ```json
  {"v":"42","s":{"type":"vehicle_save","data":{...},"position":{...},"last_velocity":[1.0,0.0,0.0],"last_angular_velocity":[0.0,0.0,0.0]}}
  ```
  * `v` — `VehicleID` (строка глобального идентификатора)
  * `s` — непрозрачный blob состояния. Точные ключи внутри `s` определяются сервером; клиенты должны передавать blob движку игры дословно.
* **Замечание по проектированию:** этот пакет освобождён от подсказки сжатия в 400 bytes — транспортный уровень ДОЛЖЕН сжимать его автоматически, потому что он всегда > 10 KB.

#### `0x43–0x4F` — зарезервировано

### 4.6 `0x50 – 0x5F` — игроки / чат

> **Канал:** только TCP

#### `0x50` — PlayerJoin
* **Направление:** сервер → все клиенты
* **Когда:** новый игрок завершает авторизацию и синхронизацию.
* **Полезная нагрузка:**
  ```json
  {"p":5,"n":"PlayerName","r":"SCR","g":false,"u":42}
  ```
  * `p` — `PlayerID` (**динамический** идентификатор сессии; меняется при каждом переподключении)
  * `n` — отображаемое имя
  * `r` — тег роли
  * `g` — флаг гостя (bool)
  * `u` — **статический** идентификатор аккаунта (постоянный, назначается при регистрации; `-1` для гостей). Также отправляется в `PlayerIDAssign` (`0x83`) самому игроку и в записях `PlayerList` (`0x54`).

#### `0x51` — PlayerLeave
* **Направление:** сервер → все клиенты
* **Когда:** игрок корректно отключается или отваливается по таймауту.
* **Полезная нагрузка:**
  ```json
  {"p":5,"n":"PlayerName","m":"left the server"}
  ```

#### `0x52` — PlayerKick
* **Направление:** сервер → целевой клиент (и опционально рассылка администраторам)
* **Когда:** сервер или Lua-скрипт кикает игрока.
* **Полезная нагрузка:**
  ```json
  {"p":5,"r":"You were kicked for inactivity"}
  ```

#### `0x53` — PlayerRoleChange
* **Направление:** сервер → все клиенты
* **Когда:** тег роли игрока обновляется во время работы (например, повышение).
* **Полезная нагрузка:**
  ```json
  {"p":5,"r":"ADM"}
  ```

#### `0x54` — PlayerList
* **Направление:** сервер → клиент
* **Когда:** при начальной синхронизации и при любом изменении состава игроков.
* **Полезная нагрузка:**
  ```json
  {"c":5,"m":10,"l":[{"p":1,"n":"Alice","u":42},{"p":2,"n":"Bob","u":-1}]}
  ```
  * `c` — текущее количество
  * `m` — максимум слотов
  * `l` — массив объектов `{p, n, u}` (`u` = статический идентификатор аккаунта, `-1` для гостей)

#### `0x55` — ChatMessage
* **Направление:** клиент → сервер → все клиенты
* **Когда:** игрок отправляет сообщение в чат.
* **Полезная нагрузка клиента:**
  ```json
  {"m":"Hello world!"}
  ```
* **Полезная нагрузка рассылки сервера:**
  ```json
  {"f":"PlayerName","m":"Hello world!"}
  ```
  * `f` — отображаемое имя отправителя (внедряется сервером; клиенты ДОЛЖНЫ игнорировать любой `f`, предоставленный другим клиентом).
* **Замечание по проектированию:** пустые сообщения и сообщения длиннее 500 кодовых единиц UTF-8 отбрасываются. Применяется ограничение частоты.

#### `0x56` — PlayerNameSync
* **Направление:** сервер → клиент
* **Когда:** во время начальной синхронизации сообщает клиенту его собственное каноническое имя.
* **Полезная нагрузка:**
  ```json
  {"p":5,"n":"PlayerName"}
  ```

#### `0x57` — ServerMap
* **Направление:** сервер → клиент
* **Когда:** во время начального рукопожатия, перед синхронизацией автомобилей.
* **Полезная нагрузка:**
  ```json
  {"m":"/levels/GridMap/main"}
  ```

#### `0x58` — PlayerCountUpdate
* **Направление:** сервер → все клиенты
* **Когда:** лёгкое обновление, когда изменилось только количество (список имён не нужен).
* **Полезная нагрузка:**
  ```json
  {"c":5,"m":10}
  ```

#### `0x59–0x5F` — зарезервировано

### 4.7 `0x60 – 0x6F` — система / события

> **Канал:** только TCP

#### `0x60` — Ping
* **Направление:** клиент → сервер
* **Когда:** пульс / измерение задержки.
* **Полезная нагрузка:** пустая (`{}`) или отсутствует (достаточно одного идентификатора типа).

#### `0x61` — Pong
* **Направление:** сервер → клиент
* **Когда:** ответ на `0x60`.
* **Полезная нагрузка:** пустая (`{}`).

#### `0x62` — AuthSuccess
* **Направление:** сервер → клиент
* **Когда:** сразу после успешного рукопожатия версии и ключа, перед синхронизацией ресурсов.
* **Полезная нагрузка:**
  ```json
  {"p":5}
  ```
  * `p` — `PlayerID`, назначенный этому соединению.

#### `0x63` — SyncRequest
* **Направление:** клиент → сервер
* **Когда:** клиент готов получить начальное состояние мира (автомобили всех существующих игроков).
* **Полезная нагрузка:** пустая (`{}`).
* **Замечание по проектированию:** заменяет устаревший `H`.

#### `0x64` — SyncBegin
* **Направление:** сервер → клиент
* **Когда:** сервер собирается начать передачу начального списка автомобилей.
* **Полезная нагрузка:** пустая (`{}`).

#### `0x65` — SyncEnd
* **Направление:** сервер → клиент
* **Когда:** все начальные автомобили отправлены; клиент теперь может считать себя полностью синхронизированным.
* **Полезная нагрузка:** пустая (`{}`).

#### `0x66` — CustomEvent
* **Направление:** двунаправленно (клиент ↔ сервер) и сервер → клиент
* **Когда:** произвольные события уровня Lua или игры.
* **Полезная нагрузка:**
  ```json
  {"n":"myEvent","d":{"foo":42}}
  ```
  * `n` — имя события
  * `d` — данные события (любое значение JSON)
* **Замечание по проектированию:** сервер блокирует жёстко заданный список исключений — событий только для сервера (`onInit`, `onPlayerAuth`, `onVehicleSpawn`, `postVehicleSpawn`, …). Полный список см. в устаревшем `HandleEvent`.

#### `0x67` — LuaEvent
* **Направление:** сервер → клиент (или клиент → сервер для событий, определённых скриптом)
* **Когда:** отличает Lua-события движка от обычных пользовательских событий.
* **Полезная нагрузка:**
  ```json
  {"n":"onSomething","d":"string_or_json"}
  ```

#### `0x68` — RelayBroadcast
* **Направление:** клиент → сервер → все остальные клиенты
* **Когда:** клиент хочет отправить непрозрачную полезную нагрузку каждому другому игроку с минимальным разбором на сервере.
* **Полезная нагрузка:**
  ```json
  {"d":"base64_or_escaped_string"}
  ```
* **Замечание по проектированию:** заменяет устаревший `N`. Поскольку полезная нагрузка непрозрачна, сервер не инспектирует `d`; он только проверяет ограничения частоты и пересылает.

#### `0x69` — ModListRequest
* **Направление:** клиент → сервер
* **Когда:** во время синхронизации ресурсов.
* **Полезная нагрузка:** пустая (`{}`).

#### `0x6A` — ModListResponse
* **Направление:** сервер → клиент
* **Когда:** ответ на `0x69`.
* **Полезная нагрузка:**
  ```json
  {"mods":[{"n":"myMod","s":12345,"v":"1.2.3"}]}
  ```

#### `0x6B` — FileRequest
* **Направление:** клиент → сервер
* **Когда:** клиенту нужен файл мода или ресурса.
* **Полезная нагрузка:**
  ```json
  {"f":"myMod.zip"}
  ```

#### `0x6C` — FileStreamBegin
* **Направление:** сервер → клиент
* **Когда:** сервер соглашается отправить файл (заменяет устаревший `AG`).
* **Полезная нагрузка:**
  ```json
  {"f":"myMod.zip","s":5242880}
  ```
  * `s` — общий размер файла в байтах.

#### `0x6D` — FileStreamChunk
* **Направление:** сервер → клиент
* **Когда:** чанк сырых данных файла. Поскольку base64 в JSON неэффективен, реализация в продакшене МОЖЕТ отправлять сырые двоичные данные после минимального заголовка; приведённый ниже пример JSON — иллюстративный.
* **Полезная нагрузка (иллюстративный JSON):**
  ```json
  {"i":0,"d":"<base64_chunk>"}
  ```
  * `i` — индекс чанка
  * `d` — байты в кодировке base64 (в продакшене: переход на сырой двоичный формат `type_id` + индекс чанка + сырые байты).

#### `0x6E` — FileNotFound
* **Направление:** сервер → клиент
* **Когда:** запрошенный файл не существует или доступ запрещён.
* **Полезная нагрузка:** пустая (`{}`).

#### `0x6F` — ResourceSyncDone
* **Направление:** клиент → сервер
* **Когда:** клиент завершил запрос всех ресурсов.
* **Полезная нагрузка:** пустая (`{}`).

---

### 4.8 `0x70 – 0x7F` — сообщения интерфейса сервера (ядро)

* `0x70` **Notification** — сервер → клиент: временное всплывающее уведомление.
* `0x71` **Dialog** — сервер → клиент: диалог подтверждения / markdown.
* `0x72 – 0x7F` — зарезервировано для будущего использования ядром. Авторы плагинов НЕ ДОЛЖНЫ использовать идентификаторы в этом диапазоне.

---

### 4.9 `0x80 – 0xFF` — рукопожатие (до авторизации) + расширения модов / плагинов

* **Рукопожатие (ядро, `0x80 – 0x85`):** `0x80` ClientVersion (лаунчер → сервер),
  `0x82` AuthKey (лаунчер → сервер), `0x83` PlayerIDAssign (сервер → лаунчер),
  `0x84` ServerError, `0x85` ServerKick. Зарезервировано ядром; плагины НЕ ДОЛЖНЫ их использовать.
  (Магия привязки UDP `0x0E` также является частью рукопожатия — сервер → лаунчер, 64 сырых
  байта, отправляется по TCP.)
* **Расширения модов / плагинов (`0x86 – 0xFF`):** непрозрачны для ядра сервера; маршрутизируются
  по таблицам подписок или ретранслируются дословно (`0x68`). Авторы плагинов МОГУТ регистрировать
  здесь обработчики; коллизии — ответственность экосистемы плагинов. **Канал:** TCP.

## 5. Семантика State-Diff (нормативно)

### 5.1 Электрика и контроллеры
1. Каждый клиент поддерживает отображение `lastSentElectrics[Vid]`.
2. При каждом локальном обновлении автомобиля клиент вычисляет разность множеств относительно `lastSentElectrics[Vid]`.
3. Только изменившиеся ключи помещаются в объект `d`/`i` пакета `0x10` или ключи `c`/`l`/`e` пакета `0x11`.
4. Сервер проверяет владение и пересылает дословно. **Начиная с v1.1 он также сливает дифф в последний известный снимок для каждого автомобиля** (электрика, ввод, режимы трансмиссии, двигатель/зажигание, вызовы контроллеров, состояние огня). Значения он по-прежнему не интерпретирует.
5. Принимающий клиент сливает входящий объект в своё локальное состояние.
6. **Воспроизведение при позднем подключении:** когда новый клиент завершает начальный список автомобилей (`0x65`), сервер воспроизводит накопленный снимок каждого автомобиля (`0x10`/`0x11`) сразу после его спавна `0x30`, чтобы подключающийся видел текущие фары, двигатель/передачу, режимы контроллеров автомобиля и то, горит ли он, — а не только конфигурацию спавна. Деформация узлов (`n`) и сцепки (`cp`) не воспроизводятся.

### 5.2 Последовательность заморозки / разморозки

> **НЕ РЕАЛИЗОВАНО** — описывает зарезервированную схему `0x40-0x4F` (см. §4.5). Сохранено для
> справки только; текущий сервер не замораживает удалённые автомобили.

1. Сервер обнаруживает, что пара «наблюдатель — автомобиль» пересекает границу `LOD_DISTANCE` (500 m).
2. Сервер отправляет `0x40` VehicleFreeze **только этому наблюдателю** по TCP.
3. Когда пара позже снова входит в `FULL_PHYSICS_DISTANCE` (300 m), сервер отправляет:
   a. `0x41` VehicleUnfreeze
   b. `0x42` VehicleSaveState (в рамках того же сброса TCP)
4. Клиент ДОЛЖЕН применить сохранённое состояние перед разморозкой физики.
5. **Пороговые значения переходов заморозки:**
   * `> 500 m` → `Frozen`
   * `300 m – 500 m` → `LOD` (клиент может локально снижать частоту физики/обновлений)
   * `< 300 m` → `Active`

### 5.3 Полномочие владельца синхронизации и постоянство (нормативно)

У каждого автомобиля есть три роли:

* **spawner** (`o`) — игрок, создавший его. Только атрибуция; сохраняется в реестре даже после отключения спавнера (отображаемое имя сохраняется с момента спавна). Используется для учёта лимитов спавна и проверок блокировки.
* **syncOwner** (`so`) — единственный подключённый клиент, отвечающий в данный момент за передачу состояния автомобиля. Сервер — единственный авторитет по этому назначению и объявляет о каждом изменении через `0x35`.
* **driver** (водитель) — игрок, находящийся в данный момент на месте водителя (отслеживается через `0x20`/`0x21`), если есть.

**Выбор владельца синхронизации** (`pickSyncOwner`, в порядке убывания приоритета):
1. Текущий **водитель**, если подключён (физическое полномочие во время вождения).
2. **Спавнер**, если подключён (игрок предпочтительно синхронизирует собственные автомобили).
3. Иначе **наименее загруженный** подключённый клиент (наименьшее число назначенных автомобилей; ничьи разрешаются по наименьшему `PlayerID`).
4. Иначе `-1` (нет подключённых клиентов — синхронизировать некому).

Выбор пересчитывается при: спавне, входе водителя (`0x20`), выходе водителя (`0x21`), отключении текущего владельца синхронизации и при подключении клиента (балансировка нагрузки). Когда результат меняется, сервер рассылает `0x35 {v, so}`. **Балансировщик** перераспределяет только *осиротевшие* автомобили (без водителя, спавнер офлайн): он перемещает по одному автомобилю за раз от наиболее загруженного к наименее загруженному клиенту, пока `max − min > 1`. Автомобили, синхронизируемые своим водителем или спавнером, балансировщик никогда не перемещает.

**Профили синхронизации** (что передаёт владелец синхронизации):
* **профиль водителя** — владелец синхронизации одновременно является локальным водителем: полное состояние — `0x01` (поз/поворот/скорость + ввод) плюс диффы `0x10`/`0x11` и обновления узлов.
* **профиль хранителя** — у владельца синхронизации нет локального водителя (автомобиль назначен балансировкой): только трансформация (`0x01` поз/поворот/скорость; ввод обнулён/опущен). Потоки диффов (`0x10`/`0x11`, узлы) **не** отправляются — без водителя состояние не меняется, и получатели сохраняют свой последний известный снимок. Пока скорость ≈ 0, частота `0x01` снижена (keep-alive), снова возрастая при движении.

**Постоянство:** отключение игрока (`0x51`) **не** удаляет автомобили. Для каждого автомобиля, чьим `syncOwner` был уходящий игрок, сервер запускает `pickSyncOwner` (спавнер, если онлайн, иначе наименее загруженный) и рассылает `0x35`; затем проход балансировщика выравнивает нагрузку. Автомобили удаляются только явным `0x31` (от владельца синхронизации или спавнера) или через Lua (`NodeMP.vehicles.remove`), а `syncOwner` становится `-1`, только когда уходит последний клиент.

**Идентичность владельца и повторная привязка при переподключении:** каждый автомобиль также хранит **статический** идентификатор аккаунта своего владельца (`u`, постоянный, отдельный от идентификатора спавнера сессии `o`). Когда игрок завершает вход, сервер сканирует мир и повторно привязывает каждый принадлежащий ему автомобиль (сопоставленный по статическому идентификатору; или по **имени** спавнера для гостевых/loopback-владельцев без аккаунта) к его новому идентификатору сессии, затем запускает `pickSyncOwner`. Эффект в том, что вернувшийся владелец немедленно вновь получает свои автомобили (учёт спавна, удаления и — если другой игрок не ведёт активно — полномочие синхронизации), и изменения блокировки `0x39` снова принимаются от него. Разделение статического/динамического также предоставляется клиентам: `0x83`/`0x50`/`0x54` несут `u` (статический) наряду с `p` (динамический).

## 7. Правила безопасности и валидации

1. **Контроль полномочий синхронизации** — пакеты, несущие состояние (`0x01`, `0x10`, `0x11`, `0x32`, `0x33`, `0x34`), принимаются **только от текущего владельца синхронизации автомобиля** (`syncOwner`). `0x31` (удаление) принимается от владельца синхронизации **или** спавнера. Пакеты занятости (`0x20`–`0x22`) несут действующего игрока и подчиняются политике сервера (блокировка проверяется по спавнеру). Пакет от любого другого клиента отбрасывается.
2. **Игнорируемые типы в UDP** — если идентификатор типа, предназначенный только для TCP, приходит по UDP, сервер ДОЛЖЕН отбросить датаграмму и опционально записать предупреждение.
3. **Ограничения частоты** — ограничения частоты на игрока применяются к чату (`0x55`), трансформациям (`0x01`), пересинхронизации (`0x37`) и пользовательским событиям / ретрансляциям (`0x66`/`0x68`); пакеты сверх лимита отбрасываются. Спавны (`0x30`) троттлятся, и PPS отслеживается.
4. **Ограничения размера** — неаутентифицированные клиенты ограничены 4 KB на TCP-пакет; аутентифицированные — 16 MB (обеспечивается проверкой заголовка длины `int32`). Диффы (`0x10`/`0x11`), пользовательские события (`0x66`) и ретрансляции (`0x68`) дополнительно ограничены 64 KiB каждый; blob состояния (`0x38`) — 4 MiB.
5. **Бомбы сжатия** — транспортный уровень ДОЛЖЕН применять `MAX_DECOMPRESSION_BUFFER_SIZE` (30 MB) и отключать клиентов, превышающих его.

## 9. Интеграция с лаунчером и маршрутизация каналов

Протокол спроектирован для работы с существующим **BeamMP Launcher** (или совместимой заменой) без изменения сетевого ядра лаунчера.

### 9.1 Логика прокси лаунчера (напоминание)
Лаунчер слушает **TCP localhost:4445** для игрового клиента и поддерживает два исходящих сокета к серверу:

| Направление | Протокол | Кадрирование |
|-----------|----------|---------|
| Лаунчер → Сервер (TCP) | `[int32 length][payload]` | Как в §2.1 |
| Лаунчер → Сервер (UDP) | `[uint8 ClientID+1][':'][payload]` | Без префикса длины |
| Сервер → Лаунчер (TCP) | `[int32 length][payload]` | Как в §2.1 |
| Сервер → Лаунчер (UDP) | сырая датаграмма | Лаунчер не снимает префикс; сервер не должен его добавлять |

Лаунчер решает **TCP или UDP** для каждого исходящего пакета с помощью жёстко заданного фильтра по первому байту:

```cpp
// Forced TCP (reliable / ack-required)
if (C == 'O' || C == 'T')  Ack = true;   // ack-required
if (C == 'N' || C == 'W' || C == 'Y' || C == 'V' || C == 'E' || C == 'C')
    Rel = true;                              // reliable
if (compressBound(size) > 1024) Rel = true; // compression heuristic
if (Ack || Rel || size > 1000)  // → TCP (SendLarge or TCPSend)
else                             // → UDP (UDPSend)
```

Поскольку NodeMP использует идентификаторы типов `uint8`, лаунчер видит новые идентификаторы как сырые байты и применяет ту же эвристику. Назначения идентификаторов типов NodeMP намеренно размещены так, что:

* **Типы только для UDP (`0x00–0x0F`)** проходят сквозь фильтр принудительного TCP и идут по UDP.
* **Все остальные типы (`0x10–0xFF`)** не начинаются с `O`, `T`, `N`, `W`, `Y`, `V`, `E` или `C` **в ASCII**, но лаунчер использует значение сырого байта. Поскольку `0x10 = 16` (управляющий символ DC1) и т. д., этих байтов НЕТ в списке принудительного TCP лаунчера. Однако, поскольку они **семантически критичны** (спавн, удаление, заморозка, чат), **сервер NodeMP ДОЛЖЕН отклонять любой пакет не из диапазона `0x00–0x0F`, приходящий по UDP**, и трактовать это как нарушение протокола.

### 9.2 Формат VehicleID
Везде, где автомобиль упоминается в сети, `v` — это **единый глобальный идентификатор**, сериализованный как десятичная строка:

```
"42"
```

* Значение — уникальное в пределах сервера целое число, выделяемое монотонно из единственного атомарного счётчика (`mNextID`).
* Оно **не** кодирует владельца; спавнер и владелец синхронизации — отдельные поля (`o`, `so`).
* Оно **никогда не переиспользуется** в рамках серверной сессии, поэтому идентичность автомобиля стабильна, даже если игрок отключается, а другой игрок позже переиспользует этот `PlayerID`.

**Примеры:**
* `"0"` — первый автомобиль, заспавненный в этой сессии.
* `"42"` — 43-й выделенный идентификатор.

**Правила разбора (на стороне сервера):**
1. Разобрать строку как беззнаковое целое.
2. Если разбор не удался, отбросить пакет.
3. Найти идентификатор в глобальном `VehicleRegistry`; если он неизвестен, отбросить пакет.

**Правила упаковки (на стороне сервера):**
```cpp
std::string PackVehicleID(VehicleID id) {
    return std::to_string(id);
}
```

**Обоснование:** глобальный идентификатор отделяет идентичность автомобиля от идентификаторов игроков, что и делает возможными **постоянство** автомобилей и **передачу владельца синхронизации**. Значение хранится как строка (а не число JSON), чтобы ключи таблиц оставались стабильными, а парсеры JSON никогда не применяли округление IEEE-754.

### 9.3 Известные ограничения
* `spawner` (`o`) — это `PlayerID` сессии, и он может устареть между отключением и переподключением; отображаемое имя сохраняется с момента спавна. При повторном входе сервер повторно привязывает его по сохранённому статическому идентификатору аккаунта владельца (`u`) — см. §5.3 «Идентичность владельца и повторная привязка при переподключении». Гостевые/loopback-владельцы (без идентификатора аккаунта) повторно привязываются по имени спавнера, что делается по мере возможности.
* Статический идентификатор аккаунта попадает на игровой сервер через погашение на бэкенде (идентификатор `nodemp:<id>`) для реальных соединений; при локальном (loopback) dev-соединении он берётся из подсказки `uid` в ключе авторизации лаунчера (доверенной только потому, что пир на той же машине).
* Лимит `MaxCars` считается по `spawner` в живом реестре. Осиротевшие автомобили (спавнер ушёл) ограничены только глобальным потолком; политика очистки по простою/таймауту оставлена на будущее.
* Балансировщик управляется событиями (подключение/отключение/выход) и перемещает автомобиль только тогда, когда дисбаланс превышает 1, чтобы избежать постоянной миграционной чехарды и избыточного трафика `0x35`. Клиент, которому назначено много припаркованных автомобилей, отправляет только их статическую трансформацию (дёшево), и нагрузка выравнивается между клиентами.
